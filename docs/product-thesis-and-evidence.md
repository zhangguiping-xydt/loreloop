# LoreLoop：产品论证、证据与开放边界

> 截至 2026-07-10。本文把产品定位、可复现实证、已知不足和开源口径放在同一处；
> 它不替代 [设计与实现说明](design-and-implementation.md)，后者说明系统如何工作。

## 一句话

**Reverse the lore. Guide the agent. Feed proof back.**

`Lore` 是项目中已经存在、但尚未被可靠表达的知识；`Loop` 是把它反构出来、用于真实
编码任务、再只把有证据的结果返回知识库的闭环。LoreLoop 的目标不是给代理增加一份
“记忆”，而是让项目知识在进入和返回时都经过可审计的信任边界。

## 解决的不是哪一种问题

典型的遗留项目没有可用的会话历史：代码、运行中的页面、过期文档和隐含约束已经存在，
但新接手的代理不知道它们。仅依赖会话记忆会有两个缺口：冷启动时没有可记的历史；代理
写进记忆的说法会在下一轮被当成事实。

LoreLoop 面对的是这个更具体的问题：**如何让代理在陌生项目中使用已有事实，同时不让
代理自己的未验证结论污染下一次任务。**

它不试图替代 Claude Code、Codex、代码索引、向量检索或团队文档；这些工具解决执行、
定位和协作中的不同部分。LoreLoop 负责的是它们之间的知识治理和验收闭环。

## 三段闭环，以及为什么它们有用

| 阶段 | 机制 | 解决的实际问题 | 不能保证什么 |
|---|---|---|---|
| **反构 / Reverse** | 从代码和真实页面提取断言级事实，附带源码 span、URL、快照或 Git 锚点 | 让陌生项目第一天就有可追溯的知识候选，而不是依赖聊天摘要 | LLM 提取仍会遗漏、合并或误读；所有提取结果默认 draft/unverified |
| **应用 / Apply** | BM25、CJK bigram、受限查询扩展和信任分层 context pack | 用少量相关事实引导代理，并把已背书事实与待核对参考明确分开 | 不保证任何 query 都召回正确答案；不使用 embedding，也不宣称通用 SOTA |
| **回流 / Return** | 浏览器或命令证据、HMAC 链、验收报告、chain-first harvest | 只让验收通过且可复审的结果提升信任，避免代理自证 | 通过一次验证不代表网页永远不变；代码再反构仍是 draft，仍需策展 |

三段缺一不可。只做反构会变成另一份可能过期的文档；只做检索会把不可信内容更高效地
注入提示词；只做验收而不回流则无法让项目在下一次任务中受益。

## 反构质量：已有证据与正确解读

公开、可重跑的评测位于 [`eval/`](../eval/)。它以固定 truths、禁止断言和一对一匹配
计分，避免把模糊的“看起来合理”当作质量。

当前记录的结果：

| 项目 | 结果 | 应如何解读 |
|---|---|---|
| Codex 代码反构 | Precision 1.00，Recall 1.00，14 条固定 truths | 在小型、固定 Python 夹具上没有误报或漏报 |
| Claude 代码反构 | Precision 1.00，Recall 0.71 | 复合断言会损失原子事实 recall，说明“可读总结”不等于“可用断言” |
| 多语言矩阵 | Precision 0.82，Recall 0.90 | Python、TypeScript 和混合夹具的方向性基线，不是现实项目的总体结论 |

因此可以说：反构管线具有可测的、可回归的质量门槛，并且能暴露模型的失败方式；不能说：
它已经证明对任意语言、任意规模仓库都高质量。源码定位、严格输出验证、一次修复尝试和
born-draft 是针对这种不确定性的工程约束，而不是装饰。

## 知识应用与召回：已有证据与正确解读

检索不把相关性藏在不可解释的黑盒里。它使用确定性的词法排序；查询扩展只能扩大候选集，
不能把模型生成的文本直接放进任务提示词，也会写入 run trace 以便审计。

当前固定检索集的结果：

| 变体 | Hit@5 | MRR | 解读 |
|---|---:|---:|---|
| 纯 BM25 | 0.50 | 0.42 | 同义词、双语表达和标识符差异会造成漏召回 |
| 固化查询扩展 | 1.00 | 1.00 | 在该固定夹具上补足了这些表达差异；不是开放域泛化声明 |
| 10k 条目 / 5 项目合成规模 | Recall@5 1.00，MRR 1.00，P95 659 ms | 说明当前实现可用但并不“瞬时”；更大语料需要持久化词法索引 |

Context pack 还有第二道判断：即使召回命中，只有当前内容 digest 被证据链背书且锚点未
漂移的事实才作为 constraint；draft、未验证或漂移条目只是 reference，rejected 和
superseded 条目不注入。召回“找到了”与知识“值得相信”是两个独立问题。

## 回流：为什么不会成为代理自我强化

回流流程是：

1. `run` 记录任务、注入条目和基线提交；代理完成工作不构成验收。
2. 操作者通过 `verify`、受限动作脚本或 `check --command` 产生可重放证据。
3. `report` 依据链、artifact、时间顺序和唯一 completion 给出 ACCEPTED/拒绝结论。
4. 只有 ACCEPTED run 能执行 `harvest`；验收断言可铸造为 verified knowledge。
5. 代码变化会再次反构，但这些 LLM 产物仍以 draft 身份进入人工策展流程。

HMAC key 位于项目树外，SQLite 和 run trace 都不是信任权威。链先于数据库写入：中断时
宁可留下 draft 或可恢复材料，也不留下没有链背书的 strong 条目。完整威胁模型和例外见
[安全策略](../SECURITY.md) 与 [设计文档](design-and-implementation.md)。

## 编码任务：产品价值的最接近证据

在同一仓库、同一任务和隐藏 contract 下，checked-in Claude 原始结果的四组对照为：
无记忆 0/3、代码索引 0/3、会话记忆 3/3、LoreLoop 3/3。过去摘要中还出现过 Codex
无知识/LoreLoop 的 0/3→3/3 对照，但仓库没有对应 raw task result 文件，因此当前自动
生成摘要和公开竞争力结论都不再引用该数字。

这是“在给定夹具中，关键项目事实被正确提供时能改变可执行结果”的证据；不是模型或
产品的全面胜负结论。隐藏 evaluator、固定输入和公开差异都保留在 `eval/`，使结论可以
被推翻或重跑，而不是营销数字。

## 相比常见方案，差异在哪里

| 常见方案 | 擅长什么 | LoreLoop 的补充，而非替代 |
|---|---|---|
| 会话记忆 | 保存本次协作形成的偏好、决策和短期上下文 | 从已有项目冷启动；不让代理直接授信自己的记忆 |
| 代码索引 / RAG | 定位相关文件和代码片段 | 输出带来源、信任等级、漂移判断和验收回流的断言级知识 |
| Agent wrapper | 统一调度模型与工具 | 把验收裁决留在链和操作者手中，而不是采信 agent 自述 |
| 团队文档 | 承载人类解释和长期约定 | 把可验证的项目事实转为可检索、可审计、可退役的条目 |

竞争力不应建立在“又一个 RAG”上，而在于可验证的知识生命周期：从事实候选，到受控应用，
再到有证据的回流。这个定位只有在真实项目的可靠性、易用性和证据规模持续提高时才成立。

## 开源与发布口径

当前适合以 **early alpha** 身份开源：本地状态、无账号/遥测、MIT 许可证、可复现 eval、
跨平台 CI、离线 demo、明确的安全边界和失败恢复说明均已具备。这里的“本地”不代表
推理一定本地；支持的宿主 CLI 仍可能把源码、页面和 prompt 发送给外部 provider。

但不应以稳定生产基础设施或模型性能冠军的口径发布。公开材料应同时保留以下事实：

- 评测夹具较小；真实公开仓库和真实使用者数据仍不足。
- 零背景可用性研究只有协议，尚无伪造的参与者结果。
- 10k 合成语料的延迟可用但不是无限扩展承诺。
- 威胁模型是 honest workstation；它不防御本机 root、恶意 agent 二进制或能读取树外 key 的攻击者。
- Agent 权限配置与进程 marker 是 cooperative defense-in-depth,不是 OS 级沙箱。
- 接口仍可能变化；升级、迁移和发布应遵循 alpha 预期。

## 下一步质量门槛

在扩大宣传或向稳定版推进前，应优先完成：

1. 至少若干真实公开仓库上的反构、检索和编码任务复现，并公开失败样本。
2. 按 `docs/usability-study.md` 完成真实零背景用户测试，记录失败、放弃和修复迭代。
3. 在超过 10k 条目的真实语料上验证延迟、上下文成本和漂移处理，再决定是否加入持久化索引。
4. 在实际 GitHub 远端/PyPI 项目上配置 protected environment 与 Trusted Publisher,
   运行已提交的 release checklist、SBOM、provenance 和依赖审计流程。

这份文档的原则是：每个“有用”的主张都应该能指向实现、测试、评测或明确的尚未证明之处。
